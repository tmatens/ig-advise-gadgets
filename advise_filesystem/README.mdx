The `advise_filesystem` gadget derives a **read-only root filesystem plus the
directories a container actually writes to**. It traces write-intent
`open`/`openat` calls and, per container (mount namespace), unions the parent
directories of the files opened for writing, then renders a Kubernetes
`securityContext` with `readOnlyRootFilesystem: true` and the writable paths the
workload needs.

It is an *advisor*, not a tracer: it emits one aggregated recommendation per
container rather than one event per open. Mapping the writable paths to the
target form (Compose `tmpfs:`, Kubernetes `emptyDir`, or a persistent volume)
and volume-vs-tmpfs correlation are left to downstream tooling.

## Requirements

- A BTF-enabled kernel (`CONFIG_DEBUG_INFO_BTF=y`) — the same requirement as the
  core `trace_open` tracer this gadget derives from.
- Root, for loading eBPF (`CAP_BPF` + `CAP_PERFMON`).

## Getting started

```bash
sudo ig run ghcr.io/tmatens/advise_filesystem:latest --containername my-app
```

## Flags

This gadget has no gadget-specific flags. Use the standard `ig`
container-selection flags (`--containername`, `--timeout`, …).

## Guide

Run a container that writes to `/var/lib/app`:

```bash
docker run -d --name demo --rm busybox:latest \
  sh -c 'mkdir -p /var/lib/app; while true; do echo x > /var/lib/app/data.txt; sleep 0.2; done'
```

Observe it:

```bash
sudo ig run ghcr.io/tmatens/advise_filesystem:latest --containername demo --timeout 8
```

The recommendation is a read-only rootfs with the one directory the workload
wrote to carved out:

```yaml
# demo
securityContext:
  readOnlyRootFilesystem: true
# writable directories the workload needs; the front-end maps these to
# tmpfs / emptyDir, or a persistent volume where a path must survive:
writable_paths:
  - /var/lib/app
```

## Limitations

- Records only write-intent opens that *succeeded*; a path the workload never
  wrote to in the window is invisible (dynamic-observation floor — a signal, not
  a proof). Grade confidence before enforcing.
- tmpfs is derived at directory granularity. Volume-vs-tmpfs correlation (does a
  written dir belong to a mount?) is left to downstream tooling, which can see
  the container's mounts. Note a raw `writable_paths` entry is not directly safe
  to apply as tmpfs — e.g. tmpfs over `/etc` would shadow the image's `/etc`.
- Written paths are workload-controlled and may contain newlines or YAML
  metacharacters, so they are emitted through a YAML-scalar escaper: a hostile
  path cannot inject a top-level key into the advice.
- If an eBPF map fills during the run, dropped observations are counted and a
  `# WARNING` comment is appended to the advice.
